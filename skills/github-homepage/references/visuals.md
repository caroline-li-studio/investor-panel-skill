# Visuals

Use visuals to make the promise visible.

## Hero Image

A hero image is useful when the project is abstract, conceptual, or social-media-promoted.

Good hero images:

- Show the main value proposition.
- Include the project name or recognizable identity when useful.
- Have enough width for GitHub README rendering.
- Avoid tiny text that will be unreadable on mobile.

For README hero images, a wide image around 1600-2400 px wide is usually fine. Keep file size reasonable, preferably under 3 MB unless visual quality requires more.

## Logo

Use a logo when the repo needs identity. Keep it simple and recognizable at small size.

## Diagrams

Use Mermaid for process and architecture:

- 4-7 nodes is usually enough.
- Use labels that describe user-facing stages.
- Keep implementation detail out of the first diagram.

## Asset Rules

- Store README assets under `assets/`.
- Use stable names such as `assets/hero.png`, `assets/logo.png`, or `assets/project-overview.svg`.
- Use relative Markdown links from README.
- Remove unused old hero assets when replacing them.

## Visual Fit Check

Ask:

- Does the first image show the selling point?
- Does it look like the target audience's world?
- Can someone understand it without reading the whole README?
- Does it avoid misleading claims?
